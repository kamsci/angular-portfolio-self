This project was a group project inspired by a real problem we experienced at General Assembly. Our Outcomes department is there to support students as they prepare for their re-entry into the workforce. They set deadlines for LinkedIn profiles, resumes, brand statements, and portfolios to be completed by. The tool we used to track these items was a Google doc.

A few of us decided that together we could create an online version of the Outcomes google doc and make a site that is easier for students to read and comment on each others work, and easier for the Outcomes team to set deliverables.

The Approach

WHAT

Fortunately for us, this project already had our user stories and requirements laid out in the form of google docs the Outcomes team already created. We have a basic set up where an admin user can login to create and manage multiple cohorts of students as well as communicate important deadlines and information with all cohorts.

Some cool features are:

  • Use of 6 normalized tables that connect to each other with foreign keys when necessary and one join table.
  • Administrators can create new users and delegate permissions for administrators
  • Administrators can add networking events they recommend to the staff picks page as well as create and set due dates for deliverables.
  • Students have one easy place to post their profile content and can view and comment on other students' profile content.
  • Students can edit their profile and reset their password

WHY

The group wanted to get expereince with something that is very common on any website that is also a product: Creating different users and assigning content and page permissions to users by their authorization; creating and querying multple databases; working on several different pages as a group with group commits.

HOW

Rails provided our backend with the MVC structure and we used the Rails controllers to direct data to their respective views. We used the rails authentication method as well as custom methods to determine authorization on each route.

We spent a good amount of time planning our databases and making sure we weren't repeating data across tables. We used the rails version of SQL to filter our data down to the cohort, student, or admin data we needed for each page. The Rails ORM was used to validate form data and set up one-to-one, one-to-many, and many-to-many relationships in the database.

The project started as a devition between front-end and back-end, but in the end, the three of us took front and back end responsibility for specific sections of the site. We colaborated on design and standarization to make our pages match. Gihub was used to manage our work and reduce bugs from code merges.
